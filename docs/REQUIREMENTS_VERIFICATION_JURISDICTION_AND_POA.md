# Requirements Verification: Jurisdiction Logic & Jurisdictional POA Wrap

This document checks implementation against **§9 Jurisdiction Logic** and **§6 The Jurisdictional POA "Wrap"**.

---

## 9. Jurisdiction Logic

| Requirement | Status | Implementation notes |
|-------------|--------|----------------------|
| **Lookup table: Zip → State, Statute Citations, Max Guest Stay** | ✅ Implemented | `jurisdictions`, `jurisdiction_statutes`, `jurisdiction_zip_mappings`. Zip → region via `jurisdiction_zip_mappings`; region → state name, statutes, `max_stay_days` via `jurisdictions` + `jurisdiction_statutes`. Service: `get_jurisdiction_for_zip()` / `get_jurisdiction_for_property()`. |
| **Zip code range (e.g. 32000–34999 → FL)** | ⚠️ Partial | Current design uses **individual 5-digit zips** in `jurisdiction_zip_mappings` (e.g. 33101→FL, 32801→FL), not numeric ranges. To support true “zip code ranges” you’d add range columns or a range check in the service. |
| **Statute text stored verbatim—no AI summarization** | ✅ Implemented | Statute **citations** stored verbatim in `jurisdiction_statutes.citation` (e.g. "F.S. § 82.036"). Optional `plain_english` is human-written in seed data; no AI is used to generate or interpret law anywhere in the flow. |
| **Used in guest acknowledgments and evidence summaries** | ✅ Implemented | **Guest acknowledgments:** Invitation/guest agreement is built from JurisdictionInfo in `build_invitation_agreement()` (statutes, removal text, max stay). **Evidence summaries:** Live property page and API return `jurisdiction_wrap` (state name, applicable statutes, removal text) and property identifier (Tax ID/APN). JLE and dashboard use jurisdiction/statutes for classification and display. |

**Relevant code:** `app/services/jurisdiction_sot.py`, `app/models/jurisdiction.py`, `app/services/agreements.py` (invitation agreement), `app/routers/public.py` (live page payload), `app/seed.py` (jurisdiction + statute seed).

## 6. The Jurisdictional POA "Wrap"

### Logic

| Requirement | Status | Implementation notes |
|-------------|--------|----------------------|
| **System is Governing Authority for the property** | ✅ | Live page presents DocuStay as operating under the owner’s granted documentation authority; records are immutable and append-only. |
| **Master POA “wrapped” with jurisdiction (RCWs/statutes by zip) + Property Tax ID/APN** | ✅ | **Master POA:** One per owner (signed at onboarding); linked from live page (“View POA”). **Jurisdiction wrap:** Built from zip/region via `get_jurisdiction_for_property()`; returned as `jurisdiction_wrap` (state name, applicable statutes, removal text). **Property identifier:** `tax_id` and `apn` on property; both returned in `LivePropertyInfo` and shown in “Property identifier” on the live page. |
| **Dynamic, legally sound package for that specific door** | ✅ | Live page is generated per request: jurisdiction by property zip/region, statutes from DB, current status, logs. No static document; content is dynamic by property. |

### Assumptions & answers

| Question | Requirement answer | Implementation |
|----------|--------------------|-----------------|
| **Master POA: one global or per-owner/per-property?** | One per Owner Entity; all properties under that owner governed by that Master POA. | ✅ One `OwnerPOASignature` per user (owner); live page resolves POA by property’s owner (profile → user_id → POA). |
| **Source of jurisdiction/statutes** | Zip-based deterministic lookup table; no AI legal interpretation. | ✅ Zip → region from `jurisdiction_zip_mappings`; region → statutes from `jurisdiction_statutes` (and jurisdiction row). All deterministic; no AI. |
| **Wrap: template merge or AI/LLM?** | Template merge; statutes inserted deterministically; AI may not interpret law. | ✅ Template merge only. Statutes and removal text come from DB and are injected into the live payload and guest agreement. No LLM in this path. |
| **Which link is scanned?** | Dedicated property-level authority link; generated by system; owner/agent may share. | ✅ Property-level **live link** by slug: `#live/{slug}` (frontend) and `GET /api/public/live/{slug}` (backend). Slug is system-generated (`live_slug`). Owner/agent can share. **Note:** URL is `/live/{slug}` rather than `/property/:id/authority`; functionally equivalent (one canonical link per property). |

### Presentation (when a cop scans the link)

| Required element | Status | Where |
|------------------|--------|--------|
| Verified Owner identity | ✅ | Live page “Quick decision layer”: “Verified owner” + owner name; owner also in “Authority layer”. |
| Master POA summary | ✅ | “Authority layer”: text that property is under signed Master POA; POA signed date; owner entity name. |
| Scope of documentation authority | ✅ | “DocuStay operates under the granted documentation authority of the owner. Records are immutable and append-only.” |
| Current property status | ✅ | Occupancy status (VACANT / OCCUPIED / etc.), authorization state (ACTIVE / NONE / EXPIRED / REVOKED), token state; current guest and dates when applicable. |
| Relevant statute citations | ✅ | “Applicable law for this property ({state_name})”: list of citations + optional plain-English; removal guest/tenant text from SOT. |
| Live timestamp | ✅ | “Generated: {generated_at}” in header; “Record ID” and “Generated” in footer. |
| Link to full expandable evidence timeline | ✅ | “Condensed audit timeline” section with scrollable list of log entries (category, title, message, timestamp). Not explicitly “expandable” UI but full timeline is shown. |
| Property identifier (Tax ID/APN) | ✅ | “Property identifier” in Authority layer: address plus APN and Tax ID when present. |

### Live vs static

| Requirement | Status |
|-------------|--------|
| View is **live**, reflects real-time system state | ✅ Each load of the live page calls the API; response includes current occupancy, current guest, logs, and `generated_at` at request time. |
| Cannot be altered retroactively | ✅ Audit logs and records are append-only; no editing of past evidence in the described flow. |

**Relevant code:** `app/routers/public.py` (`get_live_property_page`, `get_live_property_poa_pdf`), `app/schemas/public.py` (`LivePropertyPagePayload`, `JurisdictionWrap`), `frontend/pages/LivePropertyPage.tsx`, `app/services/agreements.py` (Master POA content and jurisdiction clause).

---

## Summary

- **Jurisdiction logic (§9):** Zip-based lookup, verbatim statute citations, no AI, used in guest agreements and evidence (live page). **Gap:** Zip is by explicit 5-digit list, not by numeric range (e.g. 32000–34999); add range support if required.
- **Jurisdictional POA wrap (§6):** Master POA (one per owner) + jurisdiction by zip + Tax ID/APN are combined on the **live property page**. Link is property-level by slug; page shows owner, POA summary, status, statutes, timestamp, and audit timeline. All content is template-merge and deterministic from the jurisdiction SOT.
